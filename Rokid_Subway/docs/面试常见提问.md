# 面试常见提问及回答

## Q1: 请介绍一下这个项目

**回答模板**：
> 这是我为 Rokid AR 智能眼镜开发的地铁导航应用。眼镜本身只有步行、骑行和驾车导航，我新增了地铁路线功能。用户对着眼镜说"从天安门到中关村"，系统会自动识别语音，查询地铁路线，在眼镜的 HUD 上显示多条方案供选择。
>
> 技术上，语音识别用的是 Sherpa-onnx（一个开源的端侧 ASR 引擎），路线规划用高德地图的 Web API，整个 App 在眼镜上独立运行，不依赖手机端。

---

## Q2: 为什么不直接用 Rokid 系统自带的语音识别？

**回答**：
> 我首先研究了 Rokid 的系统源码，发现它的 ASR 服务（InstructService、OnLineManager）是通过 AIDL 跨进程调用的系统级服务，对第三方 App 不开放——需要系统签名权限。我考虑过通过蓝牙 HFP 协议调用手机端的语音识别，但那需要手机端 App 配合。最终选择了在眼镜本地集成开源 ASR 引擎，这样完全自主可控。

---

## Q3: 为什么从 Vosk 换到 Sherpa-onnx？

**回答**：
> 一开始用 Vosk，是因为它是最知名的开源离线 ASR。但实际测试下来，Vosk 的小模型（vosk-model-small-cn-0.22）对地铁站名的识别非常差，比如"西单"会识别成"习单"，"国贸"变成"过贸"。
>
> 然后我调研了 Sherpa-onnx，它用的是 Zipformer 模型，在 AISHELL 测试集上的 WER 只有 1.74%，比 Vosk 的 7.9% 好了四倍多。而且 int8 量化后只有 24MB，比 Vosk 的 42MB 还小。切换后，地铁站名识别准确率明显提升。

---

## Q4: 你遇到的最大技术挑战是什么？

**回答**：
> Audio HAL 的兼容性问题。Rokid 用了定制的 Android Audio HAL（基于 AGM/PAL 框架），当我用标准的 AudioSource.VOICE_RECOGNITION 创建 AudioRecord 时，底层直接返回 -19（ENODEV），AudioRecord 初始化失败。
>
> 排查过程是：先通过 logcat 看到 AGM 的错误日志，然后反编译 Rokid 系统的 AudioController.java，发现系统自己的录音服务用的是 AudioSource.MIC(1)。改成 MIC 后，AudioRecord 就正常工作了。
>
> 这个经验让我意识到，Android 设备碎片化不只是 UI 层面的，底层 HAL 也会有定制限制。

---

## Q5: 如何实现跨城市地铁查询的？

**回答**：
> 关键在于自动推断城市。我在调用高德地理编码 API 时，不只拿回坐标，还拿回了城市名。比如用户说"从武汉站到光谷广场"，对"武汉站"做 geocode 返回坐标和 city="武汉市"，对"光谷广场"也返回 city="武汉市"。然后在调用高德路线规划 API 时，把起点城市和终点城市分别传入 `city` 和 `cityd` 参数。
>
> 这样即使用户人在北京，也能查武汉的地铁路线，因为不依赖 GPS 定位。

---

## Q6: 眼镜怎么联网？

**回答**：
> 目前用手机开 WiFi 热点，眼镜通过 WiFi 连接。理论上 Rokid 有蓝牙网络代理机制（Nav_NetProxyRequest），但这是私有协议，需要系统签名，第三方 App 无法调用。实际使用中，用户开一次热点就行，后续眼镜会自动连接。

---

## Q7: 这个项目架构怎么设计的？

**回答**：
> 采用了单 App 三 Activity 的简单架构：
>
> 1. **TransitMainActivity** — 负责语音输入和路线查询，是入口
> 2. **TransitResultActivity** — 显示多条路线方案列表
> 3. **TransitDetailActivity** — 显示选中方案的详细换乘信息
>
> 数据层有两个 Service 类：
> - **VoiceRecognizer** — 封装 Sherpa-onnx 的录音和识别
> - **AmapTransitService** — 封装所有高德 API 调用
>
> Activity 之间通过 **TransitDataHolder**（一个 object 单例）传递路线数据，避免 Intent 传递大对象的序列化开销。

---

## Q8: 为什么不用 MVVM / Jetpack Compose？

**回答**：
> 两个原因：
> 1. **设备限制** — Rokid 运行 Android 10（API 29），Compose 虽然支持但性能不如原生 View，在低端设备上渲染可能有延迟
> 2. **项目规模** — 整个 App 只有 3 个页面、~1200 行代码，用 MVVM 反而过度设计。ViewBinding 已经足够保证类型安全，代码也清晰

如果项目规模扩大（比如加上实时导航、收藏路线等功能），我会考虑引入 ViewModel + StateFlow。

---

## Q9: ASR 模型是怎么加载的？会不会影响启动速度？

**回答**：
> 模型文件放在 assets 目录，App 启动时在 IO 线程加载。Sherpa-onnx 的 int8 模型只有 24MB，在 Rokid 眼镜上加载时间大约 1-2 秒。加载期间 UI 显示"正在加载语音模型…"，加载完成后才允许语音输入。
>
> 如果要优化，可以考虑：首次启动时把模型文件从 assets 解压到内部存储，后续直接从文件系统加载，速度更快。

---

## Q10: 有没有考虑过性能优化？

**回答**：
> 1. **APK 瘦身** — 只打包 `arm64-v8a` ABI（Rokid 是 arm64 芯片），APK 从 120MB 减到 55MB
> 2. **模型量化** — 用 int8 量化后的模型，体积从 50+MB 降到 24MB，推理速度也更快
> 3. **网络优化** — OkHttp 连接池复用，地理编码结果可以缓存
> 4. **内存管理** — AudioRecord 在 stop() 时及时 release()，避免内存泄漏

---

## Q11: 如果让你继续迭代，下一步做什么？

**回答**：
> 1. **热词增强** — Sherpa-onnx 支持 hotword，可以把常见地铁站名加入热词列表，进一步提升识别率
> 2. **离线路线缓存** — 把常用城市的地铁线路图缓存到本地，减少 API 调用
> 3. **实时导航** — 结合 GPS，在用户乘坐地铁时提示到站信息
> 4. **多语言** — 支持英文语音输入，方便外国游客使用
> 5. **蓝牙联网** — 研究 Rokid 的蓝牙 PAN 或 GATT 代理，摆脱 WiFi 热点依赖

---

## Q12: 你从这个项目学到了什么？

**回答**：
> 三个方面：
> 1. **不要重复造轮子，但也要敢于替换轮子** — Vosk 是第一选择，但当它不满足需求时，果断切换到 Sherpa-onnx。关键是要有评估和对比的能力
> 2. **限制条件下的工程思维** — AR 眼镜没有键盘、没有触屏、网络受限、HAL 定制化。每个限制都需要找到变通方案，这锻炼了我的问题解决能力
> 3. **系统级 debug 能力** — 通过反编译、logcat 分析、系统源码追踪，定位到 HAL 层的兼容性问题。这种从上层到底层的排查思路，在日常开发中也非常实用

## 加分项：面试官可能追问的深层问题

### "AudioRecord 和 MediaRecorder 有什么区别？"
> AudioRecord 是底层 API，直接获取 PCM 原始音频数据，适合实时处理（如 ASR）。MediaRecorder 是高层 API，直接录制成文件（如 MP3/AAC），不适合实时流式处理。

### "ONNX Runtime 了解多少？"
> Sherpa-onnx 底层用的是 ONNX Runtime，它是微软开源的推理引擎。好处是跨平台（支持 x86/ARM/GPU），支持模型量化（int8/fp16），而且对 Transformer 类模型有专门优化。

### "高德 API 的限流策略了解吗？"
> 高德 Web API 免费版每天 5000 次调用，QPS 限制 10。如果用量大，需要申请企业认证。我们这个场景每次查询约 2-3 次 API 调用（geocode + transit），正常使用完全够。

### "Coroutines 和 RxJava 比，你为什么选 Coroutines？"
> Kotlin Coroutines 是 Kotlin 原生方案，代码更简洁（没有复杂的操作符链），而且 Kotlin 项目中 IDE 支持更好。对于我们这种以 suspend 函数为主的 IO 操作场景，Coroutines 比 RxJava 更轻量、更直观。
