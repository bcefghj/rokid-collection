# 面试项目技术细节（深度版）

## 一、项目背景

Rokid AR 智能眼镜是一款 Android 10 设备，分辨率 480×640（竖屏），有触控板（前后滑动）和物理按键，没有键盘。原系统有步行/骑行/驾车导航，但没有地铁路线功能。

**我的目标**：给眼镜加上语音驱动的地铁导航。

---

## 二、系统架构分析

### 原始导航架构
- **Phone App**（手机端）：通过高德 SDK 规划路线
- **Glasses**（眼镜端）：通过蓝牙 GATT 接收导航数据，纯显示
- 通信协议：`Nav_Start`, `Nav_UpdateInfo`, `Nav_Data` 等蓝牙消息

### 我的方案选择
经过分析，我选择**绕过蓝牙通信，独立在眼镜上开发新 App**，原因：
1. 蓝牙 GATT 私有协议，没有文档，反编译代码全混淆
2. 手机端 App 是打包好的，改不了
3. 独立 App 可以完全控制 UI 和交互逻辑
4. 眼镜可以直接连 WiFi 调用 API

## 三、语音识别（ASR）— 最大技术难点

### 3.1 方案选型过程

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| Rokid 内置 ASR | 已有，无需额外开发 | 系统服务，不对第三方 App 开放 | ❌ 放弃 |
| Vosk | 开源免费，离线运行 | 中文识别率低（WER ~7.9%），地铁站名错误率高 | ❌ 替换 |
| 百度/讯飞在线 ASR | 精度高 | 需要联网，有延迟 | ❌ 不考虑 |
| **Sherpa-onnx Zipformer** | 开源免费，离线，WER 1.74%，支持热词 | 需手动集成 | ✅ 采用 |

### 3.2 Audio HAL 适配问题（面试重点！）

**问题描述**：
- Vosk 默认用 `AudioSource.VOICE_RECOGNITION(6)`
- Rokid 的定制 Audio HAL（AGM/PAL 框架）拒绝了这个 AudioSource，返回 `-19 (ENODEV)`

**排查过程**：
1. 查看 logcat，发现 `AGM: session_connect_aif: Error:-19 graph open failed`
2. 反编译 Rokid 系统的 `AudioController.java`，发现系统自己用 `AudioSource.MIC(1)`
3. 修改代码，将 AudioSource 从 `VOICE_RECOGNITION(6)` 改为 `MIC(1)`

**代码修改**：
```kotlin
// ❌ 之前（Vosk 默认）
AudioRecord(MediaRecorder.AudioSource.VOICE_RECOGNITION, 16000, ...)

// ✅ 之后（适配 Rokid HAL）
AudioRecord(MediaRecorder.AudioSource.MIC, 16000, ...)
```

**面试话术**：
> "Rokid 使用了定制的 Audio HAL（AGM/PAL），它对 AudioSource 有白名单限制。我通过反编译系统代码，发现系统自己的录音服务用的是 MIC(1)，于是把 AudioSource 改成了 MIC，解决了麦克风无法初始化的问题。"

### 3.3 Sherpa-onnx 集成要点

- 模型：`sherpa-onnx-streaming-zipformer-zh-14M` (int8 量化，24MB)
- 采样率：16kHz mono PCM16
- 端点检测（Endpoint Detection）：
  - Rule1: 说完后 2.4 秒静音 → 结束
  - Rule2: 检测到文字后 1.4 秒静音 → 结束
  - Rule3: 最大录音 20 秒
- 流式识别：每 100ms 读取一次音频 → 喂给模型 → 实时返回部分结果

## 四、路线规划

### 4.1 高德 Web API 调用流程

```
用户说话 → ASR → "从天安门到中关村"
                    ↓ 正则解析
              origin="天安门", dest="中关村"
                    ↓ 地理编码
              geocodeFull("天安门") → 116.397,39.909 + city="北京市"
              geocodeFull("中关村") → 116.316,39.983 + city="北京市"
                    ↓ 路线规划 API
              /v3/direction/transit/integrated
              → 返回多条路线方案
```

### 4.2 跨城市支持

核心逻辑：geocodeFull 同时返回坐标和城市名。

```kotlin
data class GeocodeResult(val location: String, val city: String)
```

当用户说"从武汉站到光谷广场"：
1. geocodeFull("武汉站") → `location="114.37,30.60"`, `city="武汉市"`
2. geocodeFull("光谷广场") → `location="114.41,30.51"`, `city="武汉市"`
3. API 调用时传 `city=武汉市` 和 `cityd=武汉市`

### 4.3 自然语言解析

```kotlin
val biPatterns = listOf(
    Regex("从(.+?)到(.+)"),   // "从天安门到中关村"
    Regex("(.+?)到(.+)"),     // "天安门到中关村"
)
val destPatterns = listOf(
    Regex("去(.+)"),          // "去中关村"
    Regex("地铁去(.+)"),       // "地铁去中关村"
    Regex("到(.+)"),          // "到中关村"
)
```

## 五、UI 适配

### 5.1 AR 眼镜显示特点
- 分辨率 480×640（竖屏，与手机方向一致但更窄）
- 半透明显示（叠加在真实视野上），所以用黑底绿字
- 字号用 `px`，不是 `sp`（AR 眼镜的 DPI 跟手机不同）

### 5.2 交互映射

| 触控板动作 | Android KeyCode | 功能 |
|-----------|----------------|------|
| 前滑 | KEYCODE_DPAD_RIGHT (22) | 下一个/确认/滚动 |
| 后滑 | KEYCODE_DPAD_LEFT (21) | 上一个/返回/滚动 |
| 按下 | KEYCODE_ENTER (66) / KEYCODE_DPAD_CENTER (23) | 确认 |
| 物理返回 | KEYCODE_BACK (4) | 返回 |

### 5.3 窗口标志

```kotlin
window.addFlags(6815872)
// 6815872 = FLAG_KEEP_SCREEN_ON | FLAG_TURN_SCREEN_ON | FLAG_SHOW_WHEN_LOCKED
// 确保在眼镜 HUD 模式下正确显示
```

## 六、网络方案

眼镜连接互联网的方式：
1. **手机热点** — 手机开 WiFi 热点，眼镜连热点 ✅（当前方案）
2. **蓝牙 PAN** — 理论可行但 Rokid 未暴露 API
3. **手机蓝牙代理** — Rokid 有 `Nav_NetProxyRequest` 但是私有协议

```bash
adb shell svc wifi enable
adb shell 'cmd wifi connect-network "WiFi名" wpa2 "密码"'
```
